Agent architecture mechanism study

同一模型，Agent 的差距从哪里来？

答案往往不在模型名，而在模型外面的控制回路。我们固定 DeepSeek V4 Flash、任务、工具、预算和验收规则，只替换执行架构，完成了 15 对 A/B 运行。

结论：混合控制组合改变了这组任务的完成结果与执行路径；但后验诊断也显示，原始 ReAct 失败被工具路由歧义明显放大。因此，这不是产品排名，也不是“ReAct 天生失败”的证据。

01 · Study design

只换控制器，其他条件冻结

“一对 A/B”指同一场景、同一重复编号下分别运行两套架构。两边使用相同模型、输入、工具、轮次上限和验收器；差异只在模型外的控制方式。

A

开放式 ReAct

模型看到完整工具集合，根据上一步观察自行决定下一步，也自行判断如何从错误中恢复。

B

混合控制组合

程序保存状态、限制合法动作、按错误类型恢复，并在产物通过验收后立即停止。

五个场景 正常路径前置条件瞬时 5xx限流成功后停止

这里比较的是一个控制组合，不是单变量消融。任何结果都不能直接归因给其中某一个模块。

02 · Primary evidence

控制组合改变了完成结果，也改变了调用路径

15 对运行全部进入质量比较，没有供应商或实验夹具错误导致的排除。主结果在五个冻结场景中方向一致。

03 · Post-hoc mechanism check

ReAct 从第一步就走错了工具路径

240 次模型响应都通过 JSON 和工具参数校验，所以问题不在接口兼容性。15/15 次 ReAct 首步都用远程资源工具读取本地文件，此后在资源获取、列表查询和等待之间循环，从未读取必要输入或写出产物。

04 · Engineering implication

把判断留给模型，把边界交给程序

模型适合理解意图、比较非结构化信息和处理例外；程序更适合维护权限、顺序、预算和最终验收。一个可落地的业务 Agent 可以从下面五层开始。

  1. 01

    任务合同

    先把输入、交付物、硬约束和失败条件写成机器可检查的规则。

  2. 02

    状态与动作门

    保存当前进度，只开放此刻合法的动作，避免无效探索。

  3. 03

    工具网关

    区分本地文件、远程资源和写操作，并返回可处理的错误类型。

  4. 04

    证据记录

    把来源、状态变化和产物绑定起来，避免模型只给结论不给依据。

  5. 05

    验收与停止

    用代码检查最终状态；一旦通过就锁定成功，不再继续调用工具。

调优顺序

先修验收规则和工具合同；再补状态与动作限制；然后处理错误恢复；最后才调整 prompt、planner 或模型。否则，更强的模型也只会更快地重复错误路径。

这一路径与 ReAct 的行动—观察循环、Reflexion 的反馈记忆、LATS 的轨迹搜索，以及 ToolSandbox 的有状态工具评测相互呼应；这些论文提供模块线索，并不证明任何商业产品的内部实现。

05 · Limits and next tests

证据支持系统级差异，不支持通用排名

当前证据支持

  • 在这个冻结任务族中，控制组合改变了执行路径与最终状态。
  • 工具命名空间歧义是当前 ReAct 失败的一个放大因素。
  • 仅修路由仍无法处理前置条件与限流恢复。

当前证据不能证明

  • ReAct 作为一般架构必然失败。
  • 某个单独控制模块造成了 100 个百分点差距。
  • 结果可外推到其他模型、任务或商业产品。

下一步：预先规定单变量消融，依次测试工具路由说明、动作门控、状态记录、类型化恢复和成功锁；随后增加任务家族与随机种子，区分“运行重复”与“任务泛化”。

06 · Reproducibility
