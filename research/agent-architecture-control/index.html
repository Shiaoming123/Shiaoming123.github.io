Agent Architecture Control Study

同一个模型，为什么 Agent 会跑出不同结果？

这项实验固定模型、业务任务、工具、预算和 Verifier，只替换 Agent 的执行架构。30 次运行给出了一个很鲜明、同时也很局部的结果：在这组采购任务里，控制层改变了模型实际走过的路径。

研究边界

这是 Agent 架构机制研究，不是产品排行榜。

页面不比较 Manus、Genspark 或其他商业产品，也不推断它们的内部实现。这里的数字只属于一个模型快照、一个任务家族和五类冻结场景。

Research question

我们做了什么

同一个基座模型并不等于同一个 Agent。任务如何拆解、当前能调用哪些工具、错误怎样分类、证据放在哪里、完成条件由谁确认，都由模型外的系统决定。控制逻辑一变，模型下一轮看到的状态也会变。

我们用同一份采购决策任务搭了两套执行器。架构 A 是开放式 ReAct；架构 B 加入显式状态、动作门控、类型化恢复、证据账本和成功锁。五种场景各重复三次，并交叉 A/B 顺序，总计 30 次运行。

Figure 1 · system boundary

只替换控制架构，其余条件保持一致

这不是两个 prompt 的随手比较。实验把模型、业务任务、工具、预算和最终验收冻结下来，只改变状态如何保存、动作怎样放行、错误如何恢复以及任务何时停止。

真正发生变化的五个位置

状态
ReAct 把状态主要留在对话中；混合控制把状态写入 StateStore。
动作
ReAct 每轮看到全部公共工具；混合控制用 ActionGate 过滤当前可执行动作。
错误
ReAct 自行理解错误并决定重试；混合控制先分类，再套用有限重试策略。
证据
ReAct 从上下文回忆证据；混合控制将证据字段写入 EvidenceLedger。
停止
ReAct 由模型决定何时 finish；混合控制在 Verifier 首次通过后触发 success latch。

哪些运行不会被当成架构失败

供应商或网络没有返回可用模型响应时记为 PROVIDER_ERROR；夹具、记录器或 Verifier 自身异常时记为 HARNESS_ERROR。对应 A/B 配对成对作废并完整重跑，旧记录仍保留。模型生成的无效参数、不可解析输出或工具 schema 错误则属于 Agent 结果，不排除。

任务成功

问题重构

同一个模型，为什么会跑出两种系统

把 Agent 写成「模型 + 工具」以后，很多差异会被误归到模型能力。实际执行时，模型看到什么状态、此刻能选哪些动作、错误以后走哪条分支、谁来判断完成，都会改变下一次调用。相同权重并不意味着相同系统。

更准确的比较单位是 模型 × 上下文编译 × 状态表示 × 动作门控 × 工具合同 × 恢复策略 × Verifier。其中任何一层不同，都可能把一次可恢复错误放大成热循环，也可能在模型还没说「完成」时，由程序确认任务已经做成。

这也解释了为什么这项研究不讨论 Manus、Genspark 或任何产品谁更强。我们测的是 Agent 架构怎样改变工具执行，不是产品排行榜。

机制模型

控制面管边界，数据面管动作

01

控制面

TaskSpec 把自然语言目标变成约束、验收和预算。State Store 记录当前业务阶段，PlanGraph 表达依赖；Policy Engine 与 ActionGate 根据权限、前置条件和剩余预算筛选下一步。

02

数据面

模型提出候选动作，Tool Gateway 校验 schema、幂等键和副作用，再交给运行时执行。Observation Adapter 把 HTTP、文件或浏览器输出归一成结构化观察，避免整段日志重新污染上下文。

03

证据面

EvidenceLedger 保存事实、来源和工件 hash。Verifier 逐条检查交付物与硬约束；success latch 锁存已经验证的完成态，Finalizer 只能引用已登记证据。

04

评测面

RunTrace 记录动作、错误、状态转移和验证结果。离线回放先检查控制策略，线上再做小流量灰度。失败因此能落到具体模块，而不是统一写成「模型不稳定」。

Architecture A

开放式 ReAct

模型直接从当前对话轨迹选择下一步。环境观察会写回上下文，但任务阶段、下一必需动作与停止条件主要靠模型自行维持。

Architecture B

混合控制

模型仍负责语义选择，但显式状态计算 NEXT_REQUIRED_ACTION；ActionGate 只放行当前合格动作，Verifier 决定是否进入 success latch。

数据允许我们说

在这一套冻结任务协议、同一 deepseek-v4-flash 和同一预算下，混合控制完成了更多可验证任务，并以更少的工具调用、重复动作、时延和总 token 结束。五个场景的方向一致，且本批没有 provider 或 harness 排除。

数据还不能替我们说

这不是「控制架构对所有任务都更好」的证明。样本只有 30 次，任务来自一个供应商选择流程，模型固定为 Flash non-thinking，Architecture B 还是多个机制的组合包。Wilson 与 Newcombe 区间描述当前设计的不确定性，不把这批结果自动升级为普遍定律。

机制解释与轨迹一致：开放式执行环更容易把「已经得到信息」误当成「继续搜同类信息」，而显式下一动作和动作门控把模型拉回交付物链路。不过，当前实验同时加入了状态、门控、恢复和验证，尚不能把收益单独归给其中某一个模块。

两组的 tool_calls_after_first_pass 都是 0，但含义不同。混合控制通过 Verifier 后立即停止；ReAct 从未出现首次通过，因此没有「通过后调用」可以计数。ReAct 的冗余应看 repeated_action_count 与公开轨迹，不能用这个 0 来证明它没有多余动作。

业务设计

把模型留在需要判断的位置

问题 建议归属 理由
意图、偏好、非结构化比较 模型 需要语义判断，规则很难穷举。
权限、金额、审批、不变量 Policy Engine / 业务代码 不能接受概率性违反。
流程阶段与合法转移 State Store / 工作流 需要可恢复、可测试，不能只靠对话记忆。
参数格式、枚举、算术、hash schema / 程序验证器 确定性问题让代码回答更便宜，也更稳定。
429、5xx、未知写入状态 类型化恢复策略 重试资格、次数、退避和补偿必须统一。
最终说明 EvidenceLedger + 模型 模型负责表达，只能引用已经验证的事实。

先定义完成，再讨论规划

如果 Verifier 无法执行，planner 只能把模糊目标拆成更多模糊步骤。先把交付物、硬约束和证据要求写进 TaskSpec。

动作空间随状态收缩

所有工具永久可见，会把权限和前置条件变成提示词负担。ActionGate 应先筛选，模型再在合法集合里选择。

写操作先设计幂等与补偿

发信、付款、发布和创建订单都可能在超时后留下未知外部状态。系统应先查询状态，再决定重试、撤销或交给人工。

成功是状态，不是一句话

Verifier 通过后，success latch 应阻止没有新验收目标的调用。Finalizer 再从证据账本生成用户能复核的工件。

调优方法

调优顺序应该跟故障链走

  1. 01

    先修任务合同与 Verifier

    确认「什么算完成」可由代码、工件或明确 rubric 判断。没有这一步，成功率只是语言评分。

  2. 02

    再看状态和动作资格

    重放失败轨迹，检查缺失状态、重复动作、越权候选和未满足前置条件。优先把确定性规则移出 prompt。

  3. 03

    然后处理错误与工具合同

    区分 NOT_FOUND、PRECONDITION、RATE_LIMITED、TRANSIENT 和 UNKNOWN_EXTERNAL_STATE。每类错误有独立预算与恢复路径。

  4. 04

    最后才调 prompt、planner 与模型

    当控制边界可测以后，用冻结任务做单变量实验。比较通过率、调用数、P50/P95 时延、证据保真和每次验证成功的成本。

一次只改一个主要机制。状态账本、ActionGate、typed retry、Verifier、success latch 和 evidence finalizer 应分别消融。把它们一起打开后再宣布「架构升级有效」，只能得到方向，得不到归因。

局限与下一轮

下一轮消融要把组合包拆开

这批研究的边界

  • 只有一个采购决策任务族，每个场景三次重复。
  • 只测试 deepseek-v4-flash 的 non-thinking 配置。
  • temperature=0 与固定 fixture 让同一场景的三次结果除时延外高度一致。重复主要检验可复现性，这 30 次运行不是 30 个独立自然样本。
  • 逻辑 429 和 5xx 来自可重复 fixture，不等同于真实服务的墙钟拥塞。
  • 混合控制同时改变多个模块，当前数据不能给单模块分配因果贡献。
  • Wilson 与 Newcombe 区间只描述这套冻结设计；这项小型配对 pilot 不做总体显著性宣称。

优先做的消融

  1. 只外置 State Store，保留开放动作选择，测重复动作和断点恢复。
  2. 单独启用 ActionGate，测错误工具与前置条件拒绝。
  3. 在相同状态表示下比较自由恢复与 typed retry。
  4. 分别启用 Verifier、success latch，测假完成和成功后调用。
  5. 跨任务族、模型快照和 thinking 配置复跑，再决定哪些机制具有可迁移性。

研究脉络

论文提供模块线索，业务架构仍要自己闭环

  • ReAct

    解释动作与环境观察交替的基本执行环。它没有替产品补上持久状态、权限和确定性验收。

  • Toolformer

    讨论模型何时调用工具。生产系统仍需 Tool Gateway、权限和副作用合同。

  • Reflexion

    把反馈写成可复用的语言经验，适合失败复盘，但不能替代独立事实验证。

  • CRITIC

    借助外部工具反馈修正输出，对应校验环；业务硬约束仍应由程序或人工确认。

  • LATS

    在多条候选轨迹中搜索并利用反馈，适合开放任务，也会带来额外调用和时延。

  • WebArena

    用可复现网站和最终功能状态评测浏览器 Agent，提醒我们不要只判最后一段文字。

  • OSWorld

    把桌面任务放进真实应用环境，说明 evaluator 必须读取环境结果，而非模型自报。

  • τ-bench

    评测带工具、策略和用户交互的真实业务任务，为规则遵循和最终状态评估提供参照。

  • ToolSandbox

    用有状态工具环境与会话里程碑评估交互，接近业务 Agent 的状态与工具约束问题。

区间方法

成功率区间采用 Wilson score interval；两臂成功率差采用 Newcombe hybrid Wilson-score interval。它们比小样本正态近似更合适，但不会替代跨任务复现与单变量消融。
